AHV metrics Overwatch collects

AHV monitoring needs to show what is happening at the host, VM, and hypervisor layers without losing the CVM context that makes Nutanix different from generic virtualization stacks.

Per-host metrics

Teams moving from VMware usually discover that the old monitoring assumptions do not map cleanly onto AHV.

[ paste full paragraph from live page here ]

  • • CPU utilization at aggregate and per-core levels so hot cores do not hide behind host averages.
  • • Memory utilization with balloon driver state so operators can tell healthy efficiency from real pressure.
  • • NIC throughput, packet rates, and error counts for each host-level data path.

Per-VM metrics

That is why AHV monitoring usually breaks for VMware migrations when teams try to reuse the old observability path.

[ paste full paragraph from live page here ]

  • • CVM resource allocation versus actual CPU and memory consumption on every host.
  • • vCPU utilization and CPU ready time so waiting-for-core behavior is visible before users complain.
  • • Memory assigned versus memory reclaimed by the balloon driver for each workload.

Hypervisor-level events

  • • vDisk IOPS and latency, split into read and write paths for troubleshooting.
  • • vNIC throughput per virtual adapter plus snapshot count and total snapshot size.
  • • ADS migration events so teams know when and why a VM moved.
